# Next.js 16 + Prisma 7 scaffold gotchas

Version-specific departures from older Next/Prisma where training-data defaults are wrong. All observed directly when scaffolding a Next 16 + Prisma 7 (SQLite) app in mid-2026.

`create-next-app@latest` (v16) defaults to App Router + Turbopack. The generated project ships an `AGENTS.md` warning "This is NOT the Next.js you know — read `node_modules/next/dist/docs/` before writing code." Take it seriously; the bundled docs are the source of truth for this version.

## Next.js 16 — middleware is now "Proxy"

Middleware was renamed to **Proxy** in Next 16 (same functionality, clearer name).

- File is `proxy.ts` (or `.js`), at project root or inside `src/` — same level as `app/`. One per project.
- Export a `proxy(request: NextRequest)` function (named or default) plus `config.matcher`:

```ts
import { NextResponse, type NextRequest } from "next/server";

export function proxy(request: NextRequest): NextResponse {
  // ... redirect / rewrite / next()
}

export const config = { matcher: "/admin/:path*" };
```

- **Defaults to the Node.js runtime** (since v16). The `runtime` route-segment-config option is *not available* in proxy files — setting it throws. This means you can use Node-only APIs (`node:crypto`, etc.) in the proxy, unlike the old Edge-runtime middleware.
- Old `middleware.ts` is deprecated; migrate to `proxy.ts`.

## Prisma 7 — driver adapters, generated client location, config file

Prisma 7 changed several defaults that break code written for Prisma 5/6.

**1. Client generates into your source tree, as ESM.** Default schema from `prisma init`:

```prisma
generator client {
  provider = "prisma-client"          // NOT the old "prisma-client-js"
  output   = "../src/generated/prisma"
}
```

- Import from the generated path, e.g. `import { PrismaClient } from "@/generated/prisma/client"`, not `@prisma/client`.
- Gitignore the output dir (`/src/generated/prisma`). `create-next-app` + `prisma init` adds this automatically.
- `prisma migrate dev` applies the migration but does **not** reliably (re)generate the client — even when `src/generated/` already exists, a newly-added field can be missing from the client's types (so `tsc` fails on it) until you run `prisma generate` separately. Run it after a schema change whenever the new field's types look stale, not just when the dir is absent.

**2. No bundled database driver — you must supply a driver adapter.** Install one and pass it to `PrismaClient`:

```ts
import { PrismaBetterSqlite3 } from "@prisma/adapter-better-sqlite3";
import { PrismaClient } from "@/generated/prisma/client";

const url = process.env.DATABASE_URL!;            // e.g. "file:./prisma/dev.db"
const prisma = new PrismaClient({ adapter: new PrismaBetterSqlite3({ url }) });
```

- The SQLite adapter's config takes `{ url }` (`':memory:'` or a file URL), plus optional better-sqlite3 options.
- Postgres uses `@prisma/adapter-pg` (`PrismaPg`), etc. The adapter package also pulls in the actual native driver (e.g. `better-sqlite3`).

**3. Config moved to `prisma.config.ts`** (generated by `prisma init`), not `package.json`'s `prisma` block:

```ts
import "dotenv/config";                    // REQUIRED — see below
import { defineConfig } from "prisma/config";

export default defineConfig({
  schema: "prisma/schema.prisma",
  migrations: { path: "prisma/migrations", seed: "tsx prisma/seed.ts" },
  datasource: { url: process.env.DATABASE_URL },
});
```

- **Prisma 7 no longer auto-loads `.env`.** The generated `.env` even says so. Add `import "dotenv/config"` at the top of `prisma.config.ts` (and install `dotenv` as a devDependency), or env vars come back undefined during CLI commands.
- Seeding: set `migrations.seed` (e.g. `"tsx prisma/seed.ts"`); run with `prisma db seed`. The seed script needs its own `import "dotenv/config"` too, since it runs as a standalone process.

## React 19 — no synchronous setState inside an effect

`eslint-plugin-react-hooks` (shipped with React 19 / Next 16) errors on **`react-hooks/set-state-in-effect`**: calling `setState` synchronously in a `useEffect` body triggers cascading renders. Two fixes that satisfy it:

- **Derive-and-seed in event handlers, not effects** — e.g. seed per-item default state when the items are *added* (in the upload/split handler), not in an effect that syncs to an items array.
- **Defer state changes into the effect's async callback** — schedule the work (`setTimeout`/promise) and call `setState` *inside* that callback; the effect body itself only schedules + returns the cleanup. (A debounced fetch+poll loop fits this naturally.)

Refs to `useRef` (e.g. a request token to ignore superseded async results) are fine in the effect body — the rule is specifically about `setState`.

## Running ad-hoc TS scripts via tsx

For one-off smokes and maintenance scripts run with `tsx` (e.g. `npx tsx scripts/foo.ts`), watch two traps in this kind of project:

- **CJS target → no top-level await.** esbuild/tsx transforms the script to CommonJS, so a top-level `await` throws `Top-level await is currently not supported with the "cjs" output format`. Wrap the body in an async function and call it: `async function main(){ … } main()`. (The same applies to `tsx -e "…"` snippets.)
- **`process.exit()` truncates piped stdout.** Calling `process.exit(0)` right after an `await` discards not-yet-flushed `console.log`/`console.info` output when stdout is a pipe — the script appears to produce *nothing*. Let the process exit naturally (the event loop drains after `main()` resolves), or flush before exiting.
- The `@/…` path alias resolves fine under tsx (tsconfig `paths`), so scripts can import app modules directly (`import { x } from "@/lib/…"`).
- **Smoke-testing server actions outside Next.** `revalidatePath`/`revalidateTag` throw `Invariant: static generation store missing` when a server action runs outside a request scope (e.g. called directly from a tsx script). The action's DB writes and other side effects execute *before* that trailing call, so wrap the invocation in try/catch and assert on the side effects (DB rows, logged email) — the throw at `revalidatePath` is expected, not a failure. Early-return paths (validation rejections) never reach it and return their result normally.

## Quick checklist for a fresh Next 16 + Prisma 7 + SQLite app

```
npx create-next-app@latest web --ts --tailwind --eslint --app --src-dir --turbopack --use-npm
npm i @prisma/client @prisma/adapter-better-sqlite3 zod   # adapter is a runtime dep (imported in app code)
npm i -D prisma tsx dotenv
npx prisma init --datasource-provider sqlite
# edit prisma.config.ts: add `import "dotenv/config"` + migrations.seed
npx prisma migrate dev --name init
npx prisma generate            # if src/generated/ not populated by migrate
```
